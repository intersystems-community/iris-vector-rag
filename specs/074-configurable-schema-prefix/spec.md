# Feature Specification: Configurable SQL Schema Prefix

**Feature Branch**: `074-configurable-schema-prefix`  
**Created**: 2026-06-07  
**Status**: Draft  

## Context

`SchemaManager` currently hardcodes `"RAG"` as the SQL schema prefix throughout
`iris_vector_rag/storage/schema_manager.py`. Every table reference
(`RAG.SourceDocuments`, `RAG.DocumentChunks`, `RAG.Entities`, etc.) and every
`WHERE TABLE_SCHEMA = 'RAG'` filter is a literal string. This prevents:

- Deploying multiple IVR instances in the same IRIS namespace (e.g., `PROD` and `TEST`)
- Integrating IVR into an existing application that owns its own SQL schema
- Running isolated CI environments without schema collisions

## User Scenarios & Testing

### User Story 1 — Override schema prefix via environment variable (Priority: P1)

A developer deploying IVR in a production IRIS namespace that already has an
existing `RAG` schema from another application needs to use a different prefix
(e.g., `MYAPP_RAG`).

**Why this priority**: Unblocks any multi-tenant or multi-deployment scenario.
All other stories depend on the prefix being settable at all.

**Independent Test**: Set `IRIS_SCHEMA_PREFIX=MYAPP_RAG`, call
`SchemaManager().initialize_schema()`, confirm all created tables are in schema
`MYAPP_RAG` and no tables are created in schema `RAG`.

**Acceptance Scenarios**:

1. **Given** `IRIS_SCHEMA_PREFIX=MYAPP_RAG` is set, **When** `SchemaManager` is
   instantiated and `initialize_schema()` is called, **Then** all DDL creates
   tables under `MYAPP_RAG.*` and no DDL references `RAG.*`.

2. **Given** `IRIS_SCHEMA_PREFIX` is not set, **When** `SchemaManager` is
   instantiated, **Then** the schema prefix defaults to `"RAG"` (backward
   compatible).

3. **Given** `IRIS_SCHEMA_PREFIX=TEST`, **When** `needs_migration()` is called,
   **Then** `WHERE TABLE_SCHEMA = 'TEST'` is used in all catalog queries.

---

### User Story 2 — Override schema prefix via constructor argument (Priority: P2)

A developer writing integration tests needs to pass different schema prefixes
per test class to achieve isolation without touching environment variables.

**Why this priority**: Enables parallel test execution and CI isolation.
Env-var approach (P1) is global; constructor arg is per-instance.

**Independent Test**: Instantiate `SchemaManager(schema_prefix="TESTRUN1")` and
`SchemaManager(schema_prefix="TESTRUN2")` in the same process. Verify each
targets its own schema.

**Acceptance Scenarios**:

1. **Given** no env var is set, **When** `SchemaManager(schema_prefix="TESTRUN")` is
   constructed, **Then** all generated SQL references `TESTRUN.*`.

2. **Given** `IRIS_SCHEMA_PREFIX=ENV_SCHEMA` is set **and**
   `SchemaManager(schema_prefix="OVERRIDE")` is constructed, **Then** the
   constructor argument wins and `OVERRIDE.*` is used.

---

### User Story 3 — Schema prefix reflected in config/metadata (Priority: P3)

Operators querying `RAG.SchemaMetadata` (or the new equivalent table) need to
know which prefix is active for a given deployment.

**Why this priority**: Observability. Without this, operators cannot tell which
schema prefix was used when the schema was created.

**Independent Test**: After `initialize_schema()` with `schema_prefix="MYAPP"`,
query `MYAPP.SchemaMetadata` and confirm a row records `schema_prefix = "MYAPP"`.

**Acceptance Scenarios**:

1. **Given** schema initialized with prefix `MYAPP`, **When** `SchemaMetadata`
   table is queried, **Then** a `schema_prefix` column value of `"MYAPP"` is
   present.

---

### Edge Cases

- What happens when the prefix contains characters invalid for SQL schema names
  (spaces, hyphens)? → Validate on construction, raise `ValueError` with clear
  message. Allowed: `[A-Za-z][A-Za-z0-9_]*`.
- What happens if two `SchemaManager` instances with different prefixes share
  the same IRIS connection? → Each operates independently on its own schema;
  no cross-schema interference.
- What happens when `needs_migration()` is called before `initialize_schema()`
  with a new prefix? → Returns `True` (schema not yet present), which triggers
  normal initialization flow.
- What about the class-level caches (`_schema_validation_cache`,
  `_tables_validated`)? → Cache keys must incorporate the schema prefix, or
  caches must be keyed per-prefix. Currently they would produce false hits
  across prefixes.

## Requirements

### Functional Requirements

- **FR-001**: `SchemaManager` MUST accept a `schema_prefix` constructor
  parameter (type `str`, default `"RAG"`).
- **FR-002**: `SchemaManager` MUST read the schema prefix from environment
  variable `IRIS_SCHEMA_PREFIX` when the constructor parameter is not provided.
- **FR-003**: The constructor parameter MUST take precedence over the
  environment variable.
- **FR-004**: Every SQL statement generated by `SchemaManager` MUST substitute
  the configured prefix for `"RAG"` — no hardcoded `RAG.*` references may
  remain.
- **FR-005**: `WHERE TABLE_SCHEMA = 'RAG'` in all catalog queries MUST use
  the configured prefix.
- **FR-006**: `SchemaMetadata` table MUST store the active prefix as a
  persisted configuration key.
- **FR-007**: The class-level validation caches MUST be keyed per schema prefix
  to prevent cross-prefix cache collisions.
- **FR-008**: Schema prefix MUST be validated against `[A-Za-z][A-Za-z0-9_]*`;
  invalid values MUST raise `ValueError` at construction time.
- **FR-009**: Default behavior (no env var, no constructor arg) MUST be
  identical to the current `"RAG"` behavior — zero behavior change for existing
  deployments.

### Key Entities

- **SchemaManager**: Gains a `schema_prefix: str` instance attribute. All
  internal SQL generation must reference `self.schema_prefix` rather than the
  literal `"RAG"`.
- **IRIS_SCHEMA_PREFIX**: New environment variable read at construction time.
- **SchemaMetadata table**: Gains a `schema_prefix` configuration key entry
  persisted alongside `embedding_dimension` and `schema_version`.

## Success Criteria

### Measurable Outcomes

- **SC-001**: Running the full test suite (`pytest tests/unit/ tests/contract/`)
  with `IRIS_SCHEMA_PREFIX=TESTPFX` passes with 0 failures and all tables
  created under `TESTPFX.*`.
- **SC-002**: Running the full test suite without `IRIS_SCHEMA_PREFIX` set
  passes with 0 failures and all tables created under `RAG.*` (unchanged from
  current behavior).
- **SC-003**: No occurrences of the literal string `"RAG."` or `"'RAG'"` remain
  in `schema_manager.py` after implementation (verified via `grep`).
- **SC-004**: `SchemaManager(schema_prefix="X Y")` raises `ValueError` before
  any SQL is executed.
